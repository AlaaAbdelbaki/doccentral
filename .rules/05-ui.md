# 🎨 UI Layer Rules (Atomic Design + Riverpod Boundary)

## 🧠 Core Principle

UI is **pure presentation only**.

---

## ⚛️ Atomic Design Structure

Each feature MUST follow:

```text
presentation/
  ├── pages/        → SMART widgets (stateful entry point)
  ├── widgets/      → feature components (DUMB)
  ├── components/   → reusable UI atoms/molecules
```

---

## 🧠 State Ownership Rule

ONLY the top-level page can access Riverpod.

```dart
class PatientPage extends ConsumerWidget {
  @override
  Widget build(context, ref) {
    final patients = ref.watch(patientsProvider);

    return PatientView(
      patients: patients,
      onAdd: () => ref.read(patientControllerProvider.notifier).add(),
    );
  }
}
```

---

## 🚨 STRICT RULES

### Allowed in Pages

* Riverpod access
* State reading
* Action triggering

---

### Forbidden in Child Widgets

* ref.watch / ref.read
* business logic
* repository calls
* state mutation

---

## 🧩 Child Widget Rule

Child widgets MUST be pure:

```dart
class PatientList extends StatelessWidget {
  final List<Patient> patients;
  final void Function(Patient) onTap;

  const PatientList({
    required this.patients,
    required this.onTap,
  });
}
```

---

## 🧠 Data Flow Rule

Page → Pass props → Dumb widgets

## 🧭 Platform Rule (Desktop-first + Responsive Design)

- The application is **desktop-first by design**
- It must also support **mobile screens using responsive layouts**
- UI MUST adapt using `LayoutBuilder` and screen constraints
- Do NOT create separate UI implementations per device type
- A single UI must dynamically adjust to screen size
- Breakpoints can be used, but logic must remain shared

## 📱 Responsive Design Rules

- Use LayoutBuilder for all major pages
- Use flexible widgets (Expanded, Flexible)
- Avoid hardcoded widths/heights
- Do not create separate "mobile pages" and "desktop pages"
- One screen = adaptive layout only
